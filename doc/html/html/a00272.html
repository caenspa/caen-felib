Precondition
You have to install the library before to start using it. See Installation page for details.

Compile your project

Header

In order to use the library, just include the CAEN_FELib.h header in your project:

#include <CAEN_FELib.h>
CAEN_FELib.h
CAEN FE Library.

Shared library

Windows

CAEN_FELib.lib for both x86 and x86_64 are provided in the installation path, by default at ProgramFiles%\CAEN\Digitizers\CAEN FELib\lib. You must use add it in your Visual Studio project settings. The corresponding shared library CAEN_FELib.dll is found both there and in the default system path at SystemRoot%\System32. If you are in a x86_64 system, the x86 DLL is installed in SystemRoot%\SysWOW64.

Linux

By default libCAEN_FELib.so is installed in the system library directory: just pass -lCAEN_FELib to the linker to use it.

Develop your software

Connect to device

To connect to a device, just call the CAEN_FELib_Open() that provides the handle to the digitizer node:

uint64_t dev_handle;
int ec = CAEN_FELib_Open(url, &dev_handle);
CAEN_FELib_Open
int CAEN_FELib_Open(const char *url, uint64_t *handle)
Connect to a device.

Eventually, CAEN_FELib_Close() can be used to close the connection. It also releases any allocated memory:

int ec = CAEN_FELib_Close(dev_handle);
CAEN_FELib_Close
int CAEN_FELib_Close(uint64_t handle)
Close the connection with device.
Warning
CAEN_FELib_Open() and CAEN_FELib_Close() are not thread safe and should be invoked from the same thread.

The URL argument is provided in the standard form described in RFC 3986:

[scheme]:[//authority][path][?query][#fragment]

Scheme represent the underlying library that implements the connection. Existing implementations are:

  • dig2 for the Open-FPGA Digitizers (CAEN Dig2 library is required)
  • dig1 for the first generation Digitizers (CAEN Dig1 library is required)

The URL is not case sensitive. Multiple queries can be concatenated using the & character.

The meaning of the other fields is implementation specific, as described in the next sections.

Implementation specific settings

We report here some details for the currently existing implementation. See digitizer documentation for furter details.

Dig2

CAEN Dig2 is accessed using the dig2 scheme.

The authority field is the IP or the domain name of the device. In case of IPv4 addresses it would be something like

dig2://192.0.2.1

or, using an IPv6 address:

dig2://[2001:db8::1]

For USB connections, default host name of the form caendgtz-usb-<pid> can be used, where <pid> is the serial number of the device.

dig2://caendgtz-usb-<pid>

Additionally, a reserved authority caen.internal with path /usb/<pid>, like

dig2://caen.internal/usb/<pid>

can be used for USB connections.

For ethernet connection digitizers provides a mDNS host name in the form

dig2://caendgtz-eth-<pid>

that may not work directly, because behaviour is operating system dependent: for example Linux requires .local at the end. For ethernet connection we suggest to use the IP directly, or a domain name provided by your DNS.

When deveveloping on the embedded Open ARM environment, you may use the special path /openarm on the reserved authority:

dig2://caen.internal/openarm

instead of the IP address 172.17.0.1, that is the IP address of the Docker host in the embedded environment.

Only one connection is supported per time. If called from another instance, CAEN_FELib_Open() on the same hostname forcibly disconnects the previously connected client, if present. If invoked from the same instance, nothing is done and CAEN_FELib_DeviceAlreadyOpen is returned.

The connection supports also some queries, described in the next sections.

Monitor clients

Multiple connections to the same host are supported using monitor as URL query:

dig2://<authority and path>?monitor

Monitor clients do not have access to the endpoints and to run-related commands (arm, disarm, clear and reset must be avoided from monitor clients). Up to 3 clients are allowed per time, including the standard client. If there is a client connected, only 2 monitor clients are allowed. Monitor clients are not allowed from the same instance of the principal client (CAEN_FELib_DeviceAlreadyOpen is returned).

Log

User may enable a debug logging by defining the environmental variable SPDLOG_LEVEL=<level>, where <level> can be either "off" (default), "error", "warn" or "info" to enable. Users may also use a special query

dig2://<authority and path>?log_level=<level>

to enable log only for a specific connection.

Log output file can be found at

  • appdata%\CAEN\caendig2.log on Windows
  • ~/.CAEN/caendig2.log on UNIX

Logger is implemented using spdlog. More details can be found on its documentation.

Dig1

CAEN Dig1 is accessed using the dig1 scheme.

The connection type is specified by the URI path, and the authority meaning depends on the connection type. Each of the following paths mas an enumeration of CAEN Digitizer CAEN_DGTZ_ConnectionType.

  • /usb, equivalent to CAEN_DGTZ_USB
  • /optical_link, equivalent to CAEN_DGTZ_OpticalLink
  • /usb_a4818, equivalent to CAEN_DGTZ_USB_A4818
  • /eth_v4718, equivalent to CAEN_DGTZ_ETH_V4718
  • /usb_v4718, equivalent to CAEN_DGTZ_USB_V4718

For all paths except /eth_v4718, the authority must be caen.internal, while for /eth_v4718 it is the IP address of the V4718.

Some queries are provided to specify connection parameters, where meaningful:

  • link_num=<num>, to specify the link number (or PID on A4818 and USB V4718)
  • conet_node=<num>, to specify the CONET node
  • vme_base_address=<addr>, to specify the VME base address

For example, to connect to a digitizer using the link 2 of an A3818 and a bridge V3718, at VNE base address 0x32100000, you may use something like:

dig1://caen.internal/optical_link?link_num=2&vme_base_address=0x32100000

Log

Also the log_level=<level> query and the SPDLOG_LEVEL environmental variable are supported. See the CAEN Dig2 log section for details.

Handles

Handles represent the nodes of the device. They are hierarchically deployed in a tree that can be browsed with path similar to UNIX file paths. Handle type is uint64_t.

Relative paths

Almost every function in the API takes a base handle and a relative path as argument. A null pointer passed as path behaves as an empty string. Futhermore, is the path ends with a slash, it is discarded. In conclusion, NULL, "" and "/" actually represent the same relative path. Note that only one trailing slash is discarded at most.

A child handle can be obtained with CAEN_FELib_GetHandle():

uint64_t par_handle;
const char serial_num_path[] = "/par/serialnum";
ec = CAEN_FELib_GetHandle(dev_handle, serial_num_path, &par_handle);
if (ec != CAEN_FELib_Success)
return ec;

char serial_num_path_alt[256];

The path argument is extremely useful because you can avoid to store locally the handle of each node. Indeed, these two calls to CAEN_FELib_GetValue() return the same value (assuming the parameter value is costant):

ec = CAEN_FELib_GetValue(dev_handle, serial_num_path, value);
if (ec != CAEN_FELib_Success)
return ec;
char value_alt[256];
ec = CAEN_FELib_GetValue(par_handle, NULL, value_alt);
if (ec != CAEN_FELib_Success)
return ec;
assert(strcmp(value, value_alt) == 0);

Of course, CAEN_FELib_GetHandle() returns the same handle argument when invoked with an empty path:

uint64_t par_handle_alt;
ec = CAEN_FELib_GetHandle(par_handle, NULL, &par_handle_alt);
if (ec != CAEN_FELib_Success)
return ec;
assert(par_handle == par_handle_alt);

Node properties

Each node has a name and a type. Names are usually equal to the last part of the path, while type can be any value of the CAEN_FELib_NodeType_t enumeration. They that can be retrieved simultaneously with CAEN_FELib_GetNodeProperties():

CAEN_FELib_NodeType_t type;
int ec = CAEN_FELib_GetNodeProperties(handle, NULL, name, &type);

Interact with a device

Warning
These examples describes how to use the library, and are based on VX2740 digitizer using the Scope firmware. Procedures, parameters and commands could be different on other products. Please check the documentation of your digitizer.

Configure

To configure the digitizer, the digitizer exposes a set of parameters. Some parameters refer to digitizer-level properties, while others refer to a specific channel.

For example, digitizer-level settings can be set in this way:

int ec;
char value[256];
unsigned int record_length = 1024; // in samples
snprintf(value, sizeof(value), "%u", record_length);
ec = CAEN_FELib_SetValue(dev_handle, "/par/RecordLengthS", value);
if (ec != CAEN_FELib_Success) return ec;
unsigned int pre_trigger = 100; // in samples
snprintf(value, sizeof(value), "%u", pre_trigger);
ec = CAEN_FELib_SetValue(dev_handle, "/par/PreTriggerS", value);
if (ec != CAEN_FELib_Success) return ec;
ec = CAEN_FELib_SetValue(dev_handle, "/par/AcqTriggerSource", "TestPulse");
if (ec != CAEN_FELib_Success) return ec;
unsigned int pulser_period = 10000; // in ns
snprintf(value, sizeof(value), "%u", pulser_period);
ec = CAEN_FELib_SetValue(dev_handle, "/par/TestPulsePeriod", value);
if (ec != CAEN_FELib_Success) return ec;
unsigned int pulser_width = 16; // in ns
snprintf(value, sizeof(value), "%u", pulser_width);
ec = CAEN_FELib_SetValue(dev_handle, "/par/TestPulseWidth", value);
if (ec != CAEN_FELib_Success) return ec;
CAEN_FELib_SetValue
int CAEN_FELib_SetValue(uint64_t handle, const char *path, const char *value)
Set the value of a writable node.

To set channel-level properties, parameters are found at path /ch/<ch_idx>/par/<par_name>:

for (size_t i = 0; i < 64; ++i) {
char path[256];
snprintf(path, sizeof(path), "/ch/%zu/par/ChEnable", i);
ec = CAEN_FELib_SetValue(dev_handle, path, "true");
if (ec != CAEN_FELib_Success) return ec;
}

A special syntax in path with <ch_idx_from>..<ch_idx_to> is available to simplify the set of a parameter for all channels in the inclusive range [<ch_idx_from>, <ch_idx_to>]:

double dc_offset = 50.; // in percent
snprintf(value, sizeof(value), "%f", dc_offset);
ec = CAEN_FELib_SetValue(dev_handle, "/ch/0..63/par/DCOffset", value);
if (ec != CAEN_FELib_Success) return ec;

The default configuration (i.e. the one set at startup) can be restored at any time with a reset command:

int ec = CAEN_FELib_SendCommand(dev_handle, "/cmd/reset");
CAEN_FELib_SendCommand
int CAEN_FELib_SendCommand(uint64_t handle, const char *path)
Send a command specified by a node.

Run control

The digitizers must be armed before can start any acquisition:

int ec = CAEN_FELib_SendCommand(dev_handle, "/cmd/armacquisition");

In case of start source is set to software start, this will start an acquisition run:

int ec = CAEN_FELib_SendCommand(dev_handle, "/cmd/swstartacquisition");

Similarly, to send a software stop:

int ec = CAEN_FELib_SendCommand(dev_handle, "/cmd/swstopacquisition");

and to disarm:

int ec = CAEN_FELib_SendCommand(dev_handle, "/cmd/disarmacquisition");

The arm command also clear data currently stored in the digitizer. Manual clear command can also be executed:

int ec = CAEN_FELib_SendCommand(dev_handle, "/cmd/cleardata");

Data acquisition

Endpoints are nodes that can be used to get data using CAEN_FELib_ReadData().

Each endpoint supports a set of fields, described in detail in the parameter documentation. Each field can be represented as single value, as array or as matrix. For example, the timestamp of an event will be a single value, the waveform in a DPP firmware will be an array, while a waveform in the scope firmware will be an array of arrays, i.e. a matrix.

First of all, we need to set which endpoint we would like to use:

int ec = CAEN_FELib_SetValue(dev_handle, "/endpoint/par/activeendpoint", "scope");

Data format

Since CAEN_FELib_ReadData() is a variadic function, the user should use CAEN_FELib_SetReadDataFormat() to specify which fields will be passed as arguments, in which order and with which C type. If CAEN_FELib_SetReadDataFormat() is not used, the format is the default one specified in the parameter documentation of the specific firmware in use. The data format is saved statically within in the library, so usually it is called only once before a run, even if it can be changed also during a run.

The data format is specified as a JSON containing an array of object with two fields: "name" and "type". This is an example:

[
{ "name": "NAME1", "type": "U64", "dim": 0 },
{ "name": "NAME2", "type": "U8", "dim": 2 }
]

where NAME1 and NAME2 is a field name and U64 and U8 are two of the several possible type name specifiers mapped to standard C types, described in the following table:

Type name Standard C type
U64 uint64_t
U32 uint32_t
U16 uint16_t
U8 uint8_t
I64 int64_t
I32 int32_t
I16 int16_t
I8 int8_t
CHAR char
BOOL bool/_Bool
SIZE_T size_t
PTRDIFF_T ptrdiff_t
FLOAT float
DOUBLE double
LONG DOUBLE long double

The the "dim" field represents the dimension of the field: 0 for scalar, 1 for arrays, 2 for matrices. In case of a scalar, the type to be used as variadic argument of CAEN_FELib_ReadData() is a pointer to a variable of the type specified in table. For dimensions 1 and 2 the type required is respectively an array and an array of pointers to allocated memory.

In case of array and matrix types the type must be a pointer to the first element of the array. For example, if "type": "BOOL":

  • "dim": 0 requires bool* pointing to allocated memory of size 1 * sizeof(bool)
  • "dim": 1 requires bool* pointing to allocated memory of size defined by the documentation
  • "dim": 2 requires bool** pointing to an array of pointers to allocated memory of sizes defined by the documentation
Note
"dim": 0 is assumed as default and can be omitted:
{ "name": "NAME1", "type": "U64", "dim": 0 }
is equivalent to
{ "name": "NAME1", "type": "U64" }

For example, the data format can be set on the endpoint we want to use:

ec = CAEN_FELib_SetReadDataFormat(ep_handle, " \
[ \
{ \"name\" : \"TIMESTAMP\", \"type\" : \"U64\" }, \
{ \"name\" : \"TRIGGER_ID\", \"type\" : \"U32\" }, \
{ \"name\" : \"WAVEFORM\", \"type\" : \"U16\", \"dim\" : 2 }, \
{ \"name\" : \"WAVEFORM_SIZE\", \"type\" : \"U32\", \"dim\" : 1 } \
]");

Read data

After setting the data format, the data can be read as:

const int timeout_ms = 100;
uint64_t timestamp;
uint32_t trigger_id;
uint16_t** wf = waveform;
uint32_t sizes[64];
const int ec = CAEN_FELib_ReadData(ep_handle, timeout_ms, &timestamp, &trigger_id, wf, sizes);
CAEN_FELib_ReadData
int CAEN_FELib_ReadData(uint64_t handle, int timeout,...)
Read the data provided by an endpoint node.

As we have seen, the argument of CAEN_FELib_SetReadDataFormat() is a JSON, that can be easily constructed at runtime. Unfortunately, the interfaces of CAEN_FELib_ReadData() and CAEN_FELib_ReadDataV() only allow C-style variadic argument lists that, according to C standard, cannot be built at runtime in a portable way.

Anyway, you can easily implement a wrapper of CAEN_FELib_ReadData() that relies on a argc/argv paradigm, using one of these two proposed approaches. The principle is that all variadic arguments are required to be pointers to objects, and, according to C standard, a pointer to object of any type can be implicitly converted to pointer to void, and vice versa.

Both examples are limited to argc <= 8, but can be easily extended if you require more arguments.

The first one is faster but requires more code.

static int _read_data_wrapper_argv_v0(uint64_t ep_handle, int timeout, size_t argc, void** argv) {
switch (argc) {
case 0: return CAEN_FELib_ReadData(ep_handle, timeout);
case 1: return CAEN_FELib_ReadData(ep_handle, timeout, argv[0]);
case 2: return CAEN_FELib_ReadData(ep_handle, timeout, argv[0], argv[1]);
case 3: return CAEN_FELib_ReadData(ep_handle, timeout, argv[0], argv[1], argv[2]);
case 4: return CAEN_FELib_ReadData(ep_handle, timeout, argv[0], argv[1], argv[2], argv[3]);
case 5: return CAEN_FELib_ReadData(ep_handle, timeout, argv[0], argv[1], argv[2], argv[3], argv[4]);
case 6: return CAEN_FELib_ReadData(ep_handle, timeout, argv[0], argv[1], argv[2], argv[3], argv[4], argv[5]);
case 7: return CAEN_FELib_ReadData(ep_handle, timeout, argv[0], argv[1], argv[2], argv[3], argv[4], argv[5], argv[6]);
case 8: return CAEN_FELib_ReadData(ep_handle, timeout, argv[0], argv[1], argv[2], argv[3], argv[4], argv[5], argv[6], argv[7]);
default:
fprintf(stderr, "%s failed: argc too large\n", __func__);
return CAEN_FELib_InvalidParam;
}
}

The second one is slower because it requires an additional memcpy from user argv to a local array, and then always dereferences all supported arguments, included those beyond argc. Anyway, the advantage is that it can be implemented with much less code: it is based on the principle that a variadic function is allowed to be called with more arguments than it expects: they are simply ignored.

static int _read_data_wrapper_argv_v1(uint64_t ep_handle, int timeout, size_t argc, void** argv) {
void* a[8];
if (argc > sizeof(a) / sizeof(*a)) {
fprintf(stderr, "%s failed: argc too large\n", __func__);
return CAEN_FELib_InvalidParam;
}
memcpy(a, argv, sizeof(*a) * argc);
return CAEN_FELib_ReadData(ep_handle, timeout, a[0], a[1], a[2], a[3], a[4], a[5], a[6], a[7]);
}

Using one of these wrappers the previous example would become:

const int timeout_ms = 100;
uint64_t timestamp;
uint32_t trigger_id;
uint16_t** wf = waveform;
uint32_t sizes[64];
void* argv[] = { &timestamp, &trigger_id, wf, sizes };
const int ec = _read_data_wrapper_argv_v0(ep_handle, timeout_ms, sizeof(argv) / sizeof(*argv), argv);

Read data loop

If you want to read data from the digitizer as soon as they are available, the simplest way is to implement a loop over the CAEN_FELib_ReadData() function, as also demonstrated in the provided demo.

If an event is already available in the endpoint, the function will return immediately. Otherwise, using the timeout value passed in milliseconds as argument by the user, it will block until either:

  • the next event becomes ready (CAEN_FELib_Success is returned);
  • the end-of-run is detected (CAEN_FELib_Stop is returned); or
  • the timeout expires (CAEN_FELib_Timeout is returned); or
  • an error occurs (another error value is returned).

If timeout is zero, the function will return CAEN_FELib_Timeout immediately if no data is avaliable; if timeout is -1, it will use an infinite timeout (i.e. CAEN_FELib_Timeout will never been generated).

These four return values can be combined within a loop to implement a practical acquisition process, as illustrated below in the provided code snippet:

const int timeout_ms = 100;
int ec;
uint64_t timestamp;
uint32_t trigger_id;
uint16_t** wf = waveform;
uint32_t sizes[64];
bool quit = false;
while (!quit) {
ec = CAEN_FELib_ReadData(ep_handle, timeout_ms, &timestamp, &trigger_id, wf, sizes);
switch (ec) {
case CAEN_FELib_Success:
// do something with data...
_user_defined_plot_waveform(timestamp, trigger_id, wf, sizes);
break;
case CAEN_FELib_Stop:
// returned once after the last event of a run
quit = true;
break;
case CAEN_FELib_Timeout:
// no data received in the requested time, try again
break;
default:
// other critical error, to be handled
caen_felib_example_print_error_description(ec);
quit = true;
break;
}
}

The return value CAEN_FELib_Stop assures that no more data has been acquired in the current run. Unfortunately, the end-of-run detection could be unavailable on some endpoints, like the Raw endpoint on CAEN Dig2, because it relies on the decode of data. In this cases is up to the user to detect the end-of-run: for example it might be based on the first timeout after the run has been stopped.

Warning
The end-of-run detection based on CAEN_FELib_Timeout must be implemented carefully. Consider this scenario, when using the Raw endpoint on CAEN Dig2: you stop the acquisition using the appropriate command. If you call the CAEN_FELib_ReadData() immediately after with a short timeout, it could be possible that some events, acquired just before the stop acquisition, have not yet been transmitted from the digitizer to the DAQ system because of delaies in the TCP/IP connection. Use an appropriate timeout, in this case.

Data conversion

The parameter documentation describes the available fields for each endpoint and the native type. In any case, the library provides automatic type conversion aong types of the same kind.

For example, let's suppose the field TIMESTAMP of a decoded endpoint has native type U64: a data format containing { "name": "TIMESTAMP", "type": "DOUBLE" } is still valid, and the library will perform internally a cast to the required type.

Note
Any type can be used, but dimension cannot be changed: in this case an error will be raised by CAEN_FELib_SetReadDataFormat().
Warning
Be careful to possible data loss when using narrower than the native one.
CAEN_FELib_SetReadDataFormat() and CAEN_FELib_ReadData() on the same handle are not thread safe and should be invoked from the same thread.

Check last error

In case of error, API functions return a negative value described in CAEN_FELib_ErrorCode. CAEN_FELib_GetErrorName() and CAEN_FELib_GetErrorDescription() can be used to get a string that represent respectively a short and a verbose description of the error code:

char msg[256];
int ec = CAEN_FELib_GetErrorDescription(error_code, msg);
CAEN_FELib_GetErrorDescription

Get the detailed description of an error code.

Extended information about the last error occurred can be retrieved using CAEN_FELib_GetLastError():

char msg[1024];
int ec = CAEN_FELib_GetLastError(msg);
CAEN_FELib_GetLastError
int CAEN_FELib_GetLastError(char description[1024])
Get the detailed description of the last error occurred on the current thread for a device.
Note
The message is cleared after a successful call to CAEN_FELib_GetLastError().

For example, it can be used to get a detailed message of a failed call to CAEN_FELib_Open():

uint64_t dev_handle;
int ec = CAEN_FELib_Open("an invalid path", &dev_handle);
assert(ec == CAEN_FELib_InvalidParam);
char msg[1024];

assert(ec == CAEN_FELib_Success);
fprintf(stdout, "CAEN_FELib_Open error: %s\n", msg);
Note
CAEN_FELib_GetLastError() is thread safe: each thread that uses the library has its own message buffer.
